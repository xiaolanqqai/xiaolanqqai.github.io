## 1. 项目概述

**MiniServer Panel v3.0** 是一个面向低配 VPS（≥100MB RAM）的极简远程服务器管理面板。

- **前端层**：纯静态网页部署在 GitHub Pages，手机/电脑浏览器均可访问
- **桥接层**：服务器上运行一个单文件 C 程序（< 0.5MB RAM），把 WebSocket 流量透传到 `localhost:22`
- **协议层**：纯 TypeScript 实现的 SSH-2 协议栈，运行在浏览器中

与 v2.0 相比，v3.0 彻底移除了 Chrome/Edge 扩展与 Native Messaging Host，改用**一个零依赖的 C 桥接程序**完成 WebSocket 到 TCP 的中转。手机浏览器、桌面浏览器、Edge/Chrome/Firefox/Safari 全部通用，无需安装任何插件。

### 核心特性

| 特性 | 说明 |
|------|------|
| 跨平台访问 | 任何现代浏览器（含手机）均可使用，无需安装扩展 |
| 极低占用 | C bridge 单线程 epoll，内存 < 0.5MB，适配 100MB RAM 服务器 |
| 零外部依赖 | bridge.c 内联 SHA-1/Base64，仅需 glibc + gcc 编译 |
| 一键安装 | install.sh 自动检测系统、下载、编译、启动 |
| Token 准入 | v3.4 起 Token 默认固定为 `ssh`，前端自动填写，无需手动配置；如需自定义可通过命令行参数指定 |
| 强制本地化 | bridge 仅允许连接 `localhost`，防开放代理 |
| SSH 终端 | 基于 xterm.js 的 Web 终端，支持 PTY 与窗口大小自适应 |
| 仪表盘 | 实时显示 CPU、内存、磁盘、运行时间等系统信息 |
| 文件管理 | 浏览目录、新建文件夹、重命名、删除文件/目录 |
| 站点部署 | 一键创建静态站点目录，保存部署记录 |
| 新手引导 | 侧栏内置安装引导页，含可复制的安装命令与常见问题 |
| 连接检测 | bridge 未安装时自动弹出提示，一键跳转引导页 |
| 暗色模式 | 自动跟随系统主题切换 |

### 技术亮点

- **纯浏览器端 SSH 实现**：基于 Web Crypto API 在浏览器中实现完整 SSH-2 协议栈
- **单文件 C 桥接**：bridge.c 一文件搞定 WebSocket 握手、帧解析、epoll 多路复用、TCP 转发
- **UI 与传输彻底解耦**：GitHub Pages 托管 UI，服务器只跑一个轻量桥接进程

## 4. 模块详解

### 4.1 C 桥接程序 (bridge.c)

**文件**: [bridge.c](file:///c:/Users/43879/Desktop/极简服务器面板/bridge.c)

这是 v3.0 的核心组件。一个**单文件 C 程序**，运行在服务器上，把浏览器发来的 WebSocket 流量透传到 `localhost:22`（SSH 服务）。

#### 设计原则

- **零外部依赖**：内联实现 SHA-1、Base64，不依赖 OpenSSL，仅需 glibc
- **单线程 epoll**：I/O 多路复用，一个线程处理所有连接，内存占用极低
- **强制本地化**：仅允许 bridge 连接 `localhost`，杜绝开放代理风险
- **Token 准入**：v3.4 起 Token 默认固定为 `ssh`，前端自动填写；URL 路径 `/TOKEN` 做简单防扫描

#### 编译与运行

```bash
# 在服务器（Debian）上：
gcc -O2 -o bridge bridge.c
# 运行（默认端口 8022，Token 默认 ssh）：
./bridge
# 自定义端口（Token 仍为默认 ssh）：
./bridge 8022
# 自定义端口与 Token（可选，前端需同步修改）：
./bridge 8022 mySecretToken123
```

运行后控制台输出形如：
```
[bridge] listening on :8022  token=ssh
```

#### 核心常量

| 常量 | 值 | 说明 |
|------|-----|------|
| `MAX_EVENTS` | 64 | epoll 单次返回最大事件数 |
| `BUF_SIZE` | 65536 | 单次读写缓冲区 |
| `MAX_CLIENTS` | 16 | 同时最大客户端数 |
| `WS_GUID` | `258EAFA5-...` | WebSocket 握手 GUID |

#### 关键函数

| 函数 | 说明 |
|------|------|
| `sha1_init / sha1_block / sha1_final` | 内联 SHA-1 实现，用于 WebSocket 握手签名 |
| `sha1_base64(key, out)` | 计算 `SHA1(Sec-WebSocket-Key + GUID)` 并 Base64 编码 |
| `base64_encode / base64_decode` | 内联 Base64 实现，用于二进制数据传输 |
| `handle_ws_handshake(fd, req)` | 解析 HTTP 升级请求，返回 `101 Switching Protocols` |
| `ws_parse_frame(...)` | 解析 WebSocket 帧（文本/二进制/Close/Ping/Pong） |
| `ws_send_frame(fd, data, len, opcode)` | 发送 WebSocket 帧 |
| `connect_tcp(host, port)` | 创建 TCP 连接（仅允许 localhost） |
| `handle_ws_message(client, msg)` | 解析 JSON 消息，分发 connect/data/disconnect |
| `handle_ws_data(client, data)` | 浏览器 → SSH 方向：解码 Base64 后写入 TCP |
| `handle_tcp_data(client, buf, len)` | SSH → 浏览器方向：Base64 编码后发送 WS 帧 |
| `epoll_loop()` | 主事件循环 |

#### 通信协议（与浏览器端对称）

WebSocket URL：`ws://SERVER:PORT/TOKEN`

消息格式（JSON + Base64）：

| 浏览器 → bridge | 含义 |
|----------------|------|
| `{ "type": "connect", "host": "127.0.0.1", "port": "22" }` | 创建 TCP 连接 |
| `{ "type": "data", "data": "<base64>" }` | 发送二进制数据 |
| `{ "type": "disconnect" }` | 主动断开 |

| bridge → 浏览器 | 含义 |
|----------------|------|
| `{ "type": "connected" }` | TCP 连接成功 |
| `{ "type": "data", "data": "<base64>" }` | 接收二进制数据 |
| `{ "type": "disconnected" }` | 远端断开 |
| `{ "type": "error", "message": "..." }` | 错误信息 |

#### 资源占用

- 内存：稳定 < 0.5MB（每客户端约 64KB 缓冲区）
- CPU：空闲 0%，活跃时仅 I/O 等待
- 磁盘：可执行文件约 30KB（strip 后）
- 适配 100MB RAM 的 VPS

为最小化服务器端部署负担，v3.2 提供 Bash 一键安装脚本，将下载源码、安装编译器、编译、启动四步合并为一行命令：

### 4.2 SSH 协议实现（TypeScript）

这是项目的核心模块，在浏览器中实现了完整的 SSH-2 协议栈，**零外部依赖**，仅使用 Web Crypto API。

#### 4.2.1 packet.ts — 二进制包编解码

**文件**: [web/js/ssh-bundle.js](file:///c:/Users/43879/Desktop/极简服务器面板/web/js/ssh-bundle.js)（已打包）

实现 RFC 4253 §6 定义的 SSH 二进制包格式。

**关键函数**：

| 函数 | 签名 | 说明 |
|------|------|------|
| `encodePacket` | `(payload, blockSize) => Uint8Array` | 编码为 SSH 二进制包（含长度+填充） |
| `extractPacketLength` | `(data) => number` | 从原始数据中提取包长度 |
| `writeUint32` / `writeString` / `writeMpint` / `writeNameList` | 对应写入函数 | 编码对应类型 |
| `readUint32` / `readString` / `readMpint` / `readNameList` | 对应读取函数 | 解析对应类型 |

**SSH 二进制包格式**：
```
┌──────────────┬───────────────┬─────────┬──────────────┐
│ packet_length│ padding_length│ payload │   padding    │
│   (4 bytes)  │   (1 byte)    │ (变长)   │ (4~255 bytes)│
└──────────────┴───────────────┴─────────┴──────────────┘
```

#### 4.2.2 crypto.ts — 加密原语

**文件**: [web/js/ssh-bundle.js](file:///c:/Users/43879/Desktop/极简服务器面板/web/js/ssh-bundle.js)（已打包）

基于 Web Crypto API (`globalThis.crypto.subtle`) 实现 SSH 所需的所有加密操作。

**关键函数**：

| 函数 | 说明 |
|------|------|
| `sha256(data)` | SHA-256 哈希 |
| `hmacSha256(key, data)` | HMAC-SHA256 消息认证码 |
| `aes128CtrEncrypt / aes128CtrDecrypt` | AES-128-CTR 加密/解密 |
| `incrementCtr(iv, count)` | 计数器模式 IV 递增 |
| `verifyRsaSha256(publicKeyDer, signature, data)` | RSA-SHA256 签名验证 |
| `dhGenerateKeyPair(p, g)` | Diffie-Hellman 密钥对生成 |
| `dhComputeSharedSecret(f, x, p)` | DH 共享密钥计算 |
| `deriveKeys(K, H, sessionId)` | 派生加密/MAC 密钥（RFC 4253 §7.2，A~F） |

**密钥派生方案**（RFC 4253 §7.2）：
```
初始 IV 客户端→服务端:   HASH(K || H || "A" || session_id)
加密密钥客户端→服务端:   HASH(K || H || "C" || session_id)
完整性密钥客户端→服务端: HASH(K || H || "E" || session_id)
（服务端→客户端对应 B / D / F）
```

#### 4.2.3 kex.ts — 密钥交换

**文件**: [web/js/ssh-bundle.js](file:///c:/Users/43879/Desktop/极简服务器面板/web/js/ssh-bundle.js)（已打包）

实现 `diffie-hellman-group14-sha256` 密钥交换（RFC 4253 §8）。

**协商的算法套件**：

| 类别 | 算法 |
|------|------|
| 密钥交换 | `diffie-hellman-group14-sha256` |
| 主机密钥 | `ssh-rsa`, `rsa-sha2-256`, `rsa-sha2-512` |
| 加密（双向） | `aes128-ctr` |
| MAC（双向） | `hmac-sha2-256` |
| 压缩（双向） | `none` |

**DH 参数**：RFC 3526 Group 14（2048 位 MODP），生成元 g=2。

#### 4.2.4 transport.ts — SSH 传输层

**文件**: [web/js/ssh-bundle.js](file:///c:/Users/43879/Desktop/极简服务器面板/web/js/ssh-bundle.js)（已打包）

`SSHTransport` 类管理整个 SSH 会话生命周期。

**连接流程（6 阶段）**：
```
Phase 1: 版本交换（raw bytes）→ 发送 "SSH-2.0-MiniServerPanel_1.0"
Phase 2: 切换到帧模式
Phase 3: 密钥交换（未加密）→ performKeyExchange()
Phase 4: NEWKEYS 交换
Phase 5: 激活加密（AES-128-CTR + HMAC-SHA256）
Phase 6: 用户认证（密码）
```

**公开 API**：

| 方法 | 返回值 | 说明 |
|------|--------|------|
| `connect(host, port, username, password)` | `Promise<void>` | 建立 SSH 连接 |
| `openShell(cols, rows)` | `Promise<ShellChannel>` | 打开交互式 Shell |
| `execCommand(command)` | `Promise<{stdout, stderr, exitCode}>` | 执行单条命令 |
| `openSftp()` | `Promise<SftpChannel>` | 打开 SFTP 子系统 |
| `disconnect()` | `void` | 断开连接并清理资源 |
| `isConnected()` | `boolean` | 检查连接状态 |
| `getServerVersion()` | `string` | 获取服务端 SSH 版本 |

### 4.3 WebSocket 传输层

**文件**: [web/js/ws-socket.ts](file:///c:/Users/43879/Desktop/极简服务器面板/web/js/ws-socket.ts)（编译后并入 [web/js/ssh-bundle.js](file:///c:/Users/43879/Desktop/极简服务器面板/web/js/ssh-bundle.js)）

`WsSocket` 类实现 `ITcpSocket` 接口，是浏览器与服务器 C bridge 之间的唯一通道。

**构造函数**：

```typescript
class WsSocket implements ITcpSocket {
  constructor(bridgeUrl?: string) {
    // 默认 'ws://127.0.0.1:8022/ssh'
    // 实际使用时由 app.js 传入 ws://HOST:PORT/TOKEN
  }
}
```

**关键行为**：
- `connect(host, port)`：先建立 WebSocket，握手成功后发送 `{type:"connect", host, port:String(port)}`
- `send(data)`：将 `Uint8Array` Base64 编码后发送 `{type:"data", data}`
- `onData(cb)`：接收 `{type:"data"}` 消息，Base64 解码后回调
- 连接超时 10 秒，错误提示："连接桥接服务超时，请确认服务器上 bridge 已启动且 Token 正确"

## 6. 数据流与通信协议

### 完整数据流（用户输入命令 → SSH 执行）

```
用户在 xterm.js 输入字符
    ↓
app.js: shell.write(data)
    ↓
SSHTransport: 加密为 SSH 通道数据帧
    ↓
WsSocket.send(encryptedBytes)
    ↓ Base64 编码
WebSocket 帧: { "type": "data", "data": "<base64>" }
    ↓ WebSocket 文本帧
C bridge: ws_parse_frame → handle_ws_message
    ↓ Base64 解码
bridge: write(tcp_fd, rawBytes, len)
    ↓ TCP 流
localhost:22 SSH 服务
```

### 反向数据流（SSH 输出 → 终端显示）

```
localhost:22 SSH 服务
    ↓ TCP 流
bridge: epoll_wait → handle_tcp_data
    ↓ Base64 编码
WebSocket 帧: { "type": "data", "data": "<base64>" }
    ↓ WebSocket 文本帧
WsSocket.onmessage → Base64 解码 → onData(bytes)
    ↓
SSHTransport: 解密 SSH 通道数据帧
    ↓
ShellChannel.onData → xterm.js.write()
    ↓
终端显示
```

### 连接生命周期

```
1. 浏览器：new WebSocket("ws://SERVER:PORT/TOKEN")
2. bridge：epoll 接受连接 → handle_ws_handshake（校验 Token）
3. 浏览器：WsSocket.onopen → 发送 {type:"connect", host, port}
4. bridge：connect_tcp("127.0.0.1", 22) → 返回 {type:"connected"}
5. 浏览器：SSHTransport 开始 SSH 协议握手（版本交换→KEX→认证）
6. 双向透传 SSH 加密数据流
7. 任端断开：bridge 发 {type:"disconnected"}，浏览器清理资源
```

## 8. 运行方式

### 8.1 服务器端：安装并运行 C bridge

提供两种安装方式，**推荐使用一键安装脚本**（自动完成所有步骤）。

#### 方式 A：一键安装脚本（推荐，最省事）

在服务器上执行一行命令，自动完成下载源码、安装 gcc、编译、启动：

#### 方式 B：源码编译（需 gcc）

适用于需要审查源码或手动控制每个步骤的场景。

```bash
# 1. 下载源码
wget -O bridge.c https://raw.githubusercontent.com/xiaolanqqai/xiaolanqqai.github.io/master/index/Tool/web/bridge.c

# 2. 安装 gcc（如未安装）
apt update && apt install -y gcc

# 3. 编译
gcc -O2 -o bridge bridge.c

# 4. 运行
./bridge
# 或自定义端口与 Token：
./bridge 8022 mySecretToken123
```

```bash
# 简单方式：nohup
nohup ./bridge 8022 mySecretToken123 > bridge.log 2>&1 &

# 或 systemd 服务（/etc/systemd/system/bridge.service）
[Unit]
Description=MiniServer Bridge
After=network.target

[Service]
ExecStart=/root/bridge 8022 mySecretToken123
Restart=always
User=root

[Install]
WantedBy=multi-user.target
```

#### 防火墙放行

```bash
# 放行 bridge 端口（不要放行 22，SSH 仅本机访问）
iptables -A INPUT -p tcp --dport 8022 -j ACCEPT
```

### 技术细节

#### 浏览器 ping 的实现

浏览器出于安全限制无法发送 ICMP 包，传统 ping 无法实现。本方案采用 `fetch` + `no-cors` 模式：

```javascript
fetch(`http://${host}:${port}/?_=${Date.now()}`, {
  mode: 'no-cors',
  signal: controller.signal,
  cache: 'no-store'
})
```

- `mode: 'no-cors'`：允许跨域请求，响应类型为 `opaque`（不可读 body，但能区分连接成功 vs 失败）
- TCP 三次握手成功即视为"可达"，无论后续 HTTP 层返回什么
- 端口不通时浏览器抛 `TypeError: Failed to fetch`，进入 catch 判定不可达
- 加随机参数 `?_=timestamp` 避免中间缓存

#### WebSocket 检测的关闭处理

```javascript
const finish = (ok) => {
  if (done) return  // 防止 onopen/onerror/onclose/timeout 重复触发
  done = true
  clearTimeout(timer)
  try { ws.close() } catch {}
  resolve(ok)
}
```

- 用 `done` 标志位保证只触发一次回调
- 无论成功失败都立即 `ws.close()` 释放连接
- `try/catch` 包裹 `close()` 防止 readyState 已是 CLOSING/CLOSED 时抛错

### 背景

部分廉价 VPS（如某些 NAT VPS、共享 IP VPS）没有独立公网 IP，服务器位于运营商内网（如 `10.0.0.x`）。用户访问流程为：

```
浏览器 → 运营商公网 IP:外部端口 → [运营商 NAT 转发] → 服务器内网 IP:8022 → bridge → localhost:22 (SSH)
```

v3.4.2 之前，引导页完全没提这种情况，NAT VPS 用户按引导装完 bridge、填 `服务器IP:8022` 连接会失败，且不知道原因。

### 两种 VPS 类型对比

| 类型 | 独立公网 IP VPS | NAT VPS / 共享 IP |
|------|----------------|-------------------|
| 运营商面板 | 无"端口转发"选项 | 有"端口转发/映射/NAT"选项 |
| 公网访问 | 直接 `公网IP:8022` | 需配置转发规则 |
| 服务器内网 IP | 无（公网 IP 直接绑在网卡） | 有（如 10.0.0.5） |
| 前端"服务器地址" | 服务器公网 IP | 运营商公网 IP |
| 前端"Bridge 端口" | 8022 | 运营商分配的外部端口（如 30001） |
| 是否需转发 SSH 22 端口 | 否 | 否（SSH 走 bridge 内部，不经公网） |
